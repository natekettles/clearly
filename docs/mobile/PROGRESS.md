#### Decisions Made
- **WindowGroup over DocumentGroup for Phase 4.** `IMPLEMENTATION.md` said `DocumentGroup(newDocument: { MarkdownDocument() }) { ‚Ä¶ }` + a separate `WindowGroup` for welcome. Deviated to a single `WindowGroup` hosting welcome + sidebar + detail. `DocumentGroup` opens the system document browser at launch with no first-class hook for a custom vault sidebar; the phase's "sidebar of `.md` files in the iCloud Clearly folder" criterion needs a custom vault browser, not a per-document scene. Keeps the iOS and Mac mental models aligned (Mac uses `Window` + `WorkspaceManager`, not `DocumentGroup`). `MarkdownDocument` still promoted to `FileDocument` as Phase 5 / 6 setup. If we want Files.app "open in Clearly" integration later, `DocumentGroup` can land as a second scene.
- **`FileDocument`, not `ReferenceFileDocument`.** Reference semantics matter for collaborative editing and multi-scene live-update; value semantics are enough for the editor binding we'll add in Phase 5. Phase 6 can upgrade if autosave flows need the reference model.
- **Flat file list, not a hierarchical tree.** Phase 4 shows `[VaultFile]` in a simple `List`. Subdirectories are walked (via `FileNode.buildTree`) but flattened into a single sorted list by filename. Hierarchy + `DisclosureGroup` expansion lands alongside Phase 8's index rebuild.
- **`.fileImporter` (SwiftUI) instead of direct `UIDocumentPickerViewController`.** Same underlying presenter, less UIKit glue.
- **Picked-iCloud vs local kind = a user-facing label only.** Functionally both paths create security-scoped bookmarks the same way; the distinction is stored so Settings in Phase 13 can group them differently. Not worth inferring the kind from URL path.
- **`NSMetadataQuery` only for `.defaultICloud`.** Picked iCloud folders can sit anywhere in iCloud Drive (not necessarily inside our ubiquity container), so the `NSMetadataQueryUbiquitousDocumentsScope` predicate wouldn't reliably cover them. Safer to treat all picked folders (iCloud or local) via `FileNode.buildTree` + manual refresh. Real-time iCloud updates only available when using the default container ‚Äî acceptable tradeoff for Phase 4.
- **Deferred `CloudBackend` persistence for `VaultWatcher`.** Did not build abstract `BackendProtocol` or split into two types ‚Äî a single `useCloudQuery: Bool` flag routes between the two code paths. Cheaper than polymorphism for two call sites.
- **Security-scoped bookmark options: `[]` on iOS, `.withSecurityScope` on macOS.** `.withSecurityScope` is macOS-only; passing it on iOS throws `NSFileReadNoPermissionError`. Branched with `#if os(iOS)` inside `VaultLocation.resolve`.
- **`@Observable` for `VaultSession` + `ObservableObject` for `VaultWatcher`.** `VaultSession` is consumed by SwiftUI views ‚Üí `@Observable` for property-level tracking. `VaultWatcher` is a worker class that mirrors state via Combine ‚Üí `ObservableObject` + `@Published` keeps the Combine interop idiomatic.
- **`.fullScreenCover` for welcome, not `.sheet`.** Welcome is a first-launch-or-change-vault gate; `.sheet` lets users swipe to dismiss before configuring a vault, which puts the app into a broken state. `interactiveDismissDisabled(session.currentVault == nil)` pins it when no vault exists.

Avoids a "has-this-already-been-resolved" bookkeeping table ‚Äî iCloud is the source of truth.

#### Decisions Made
- **One `IOSDocumentSession` per tab (and therefore one `NSFilePresenter` per tab), not a single shared session that rebinds on tab switch.** CLAUDE.md's `NSFilePresenter` discipline ("every `addFilePresenter` MUST be paired with `removeFilePresenter` before the presenter's owning object deallocates") and the Phase 6 per-document-not-per-vault decision both demand a 1:1 mapping between open documents and presenters. Memory cost is small; a session holds a string + a bookkeeping struct + a one-op operation queue.
- **No untitled tabs on iOS v1.** Mac has them (`createUntitledDocument` ‚Üí `Untitled 1`). iOS has no scratch folder story and doesn't write untitled markdown files anywhere, so there's no natural home for an untitled buffer. ‚åòT simply opens the quick switcher ("new tab = find or create a file") ‚Äî which reuses `VaultSession.openOrCreate` and still delivers the "add a tab" user model without inventing an iOS-only scratch buffer.
- **Tap = open-or-activate (no "replace active tab" semantics).** Plan floated "tap replaces active tab, long-press for new tab" as an option. Switched to Safari/Bear/Finder semantics: tap always opens in a new tab if the file isn't already open, activates the existing tab if it is. Long-press context menu only holds Rename + Delete. Satisfies the "Open 5 notes into 5 tabs" success criterion with the simplest possible interaction model. No pending-new-tab flag needed on the controller.
- **Per-tab `OutlineState` + `BacklinksState` (not per-view).** Live on `IPadTab` so switching tabs doesn't re-parse headings or re-query backlinks. Wired to the shared `DocumentDetailBody` as `@ObservedObject` parameters. `EditorView_iOS`'s `attachOutlineState` already handles reference changes correctly (identity check + closure re-bind), so the scroll-to-heading callback retargets cleanly on tab switch.
- **`NavigationSplitView(.balanced)`, not manual device-size branching.** iOS 17 auto-collapses columns on smaller iPads / portrait ‚Äî no `UIDevice.userInterfaceIdiom` checks needed, and the layout reacts to multitasking-size changes that static idiom wouldn't. ## Architectural Decisions

### 2026-04-20 ‚Äî Universal Purchase with shared bundle ID across all three channels
Direct (Sparkle), Mac App Store, and iOS App Store all use `com.sabotage.clearly`. Universal Purchase links the two App Store SKUs; direct distribution is outside the App Store and unaffected. On a single Mac a user installs direct OR MAS, not both (same bundle ID collision). iCloud container `iCloud.com.sabotage.clearly` provisioned once against the Team ID and shared.

### 2026-04-20 ‚Äî Direct Mac build participates in iCloud sync
Direct-download users sync with iOS via iCloud too. Costs nothing extra; avoids a two-tier "MAS users get sync, direct users don't" story.

### 2026-04-20 ‚Äî iPad multi-document tabs in v1 (research default overridden)
Research recommended single-document-per-scene on iPad to match `DocumentGroup` semantics. Overridden to port the Mac tab bar because Clearly's Mac experience is defined by tabs, and iPad hardware is similar enough that users will expect parity. Implementation keeps `DocumentGroup` as the scene root and manages multiple `OpenDocument` instances inside a custom container in the detail column.

### 2026-04-20 ‚Äî 13 phases rather than 8
First draft had 8 phases but several were 2‚Äì3 days of work each. Split to 13 phases sized so each can be completed and verified on-device in one focused session (half-day to full-day).

### 2026-04-20 ‚Äî iOS simulator verification path
Preferred verification path for the iOS target is a normal destination-based build: `xcodebuild -scheme Clearly-iOS -destination 'platform=iOS Simulator,name=iPhone 17,OS=26.4' -configuration Debug build`. `xcrun simctl install` + `xcrun simctl launch` remain useful when we want an explicit post-build launch check, but they are not required to work around destination resolution in this workspace.

### 2026-04-21 ‚Äî Per-document `NSFilePresenter`, not per-vault (Phase 6)
`NSFilePresenter` is keyed on `presentedItemURL`. Registering a single presenter for the vault folder URL would deliver `presentedItemDidChange` callbacks for every file the ubiquity daemon touches ‚Äî the wrong granularity for the "this specific open note changed remotely" use case Phase 6 needs for in-editor refresh. Vault-level change detection is still handled by `VaultWatcher`'s `NSMetadataQuery`. One caveat to revisit in Phase 12 (iPad multi-document tabs): multi-presenter lifecycle management becomes stricter when several documents are open simultaneously.

### 2026-04-21 ‚Äî `@State`-scoped `IOSDocumentSession`, not scene-level (Phase 6)
Document state is per-file and the detail view's `.task(id:)` + `.onDisappear` already drive open / close cleanly. Lifting the session to an `@Environment` value at the app scene would have required explicit file-switch teardown that the detail view gets for free. If a future "resume last note on cold launch" feature needs persisted state across view lifetime, we'd promote this ‚Äî not before.

### 2026-04-21 ‚Äî Belt-and-braces `isOwnWriteInFlight` flag (Phase 6)
`NSFileCoordinator(filePresenter:)` suppresses callbacks to the passed-in presenter for the duration of the coordinated operation ‚Äî that's the official contract. But a metadata-query change event can fire on a separate path (ubd notifying `NSMetadataQuery` of a file-size change before the coordinator's suppression has fully propagated). The `isOwnWriteInFlight` flag is a cheap additional guard against the echo. Doesn't replace presenter-aware coordination; just belts the braces.

### 2026-04-21 ‚Äî Highlighter lives on Coordinator, not UITextView (Phase 5)
First draft had `ClearlyUITextView` own a `MarkdownSyntaxHighlighter` and install itself as `NSTextStorageDelegate`. Moved the highlighter to `EditorView_iOS.Coordinator` to mirror the Mac pattern (`EditorView.Coordinator` owns the highlighter; `NSTextViewDelegate.textDidChange` drives it). Avoids the `NSTextStorageDelegate.didProcessEditing` ‚Üí nested `beginEditing`/`endEditing` re-entry risk ‚Äî `UITextViewDelegate.textViewDidChange` fires outside the text storage's internal editing transaction. `ClearlyUITextView` is now a pure-config subclass (typing attributes, insets, keyboard config only).

### 2026-04-21 ‚Äî `text` as `let`, not `@Binding`, in Phase 5 editor
Read-only contract enforced structurally rather than by convention. Phase 6 will promote to `@Binding` when writes land. The `pendingBindingUpdates` counter is already in place in the Coordinator so that promotion won't require re-architecting the cursor-jump guard.

### 2026-04-21 ‚Äî Asset-catalog-backed colors, `clearlyAsset` helper (Phase 12b.1)
Migrated all 23 dynamic colors from in-code `clearlyDynamic(name:light:dark:)` providers into `Packages/ClearlyCore/Sources/ClearlyCore/Resources/Colors.xcassets` with Any + Dark appearance variants. Rationale: unblocks future High Contrast variants via a simple Xcode appearance toggle, and lets designers tweak colors without touching Swift. Introduced `PlatformColor.clearlyAsset(named:)` helper that wraps `NSColor(named:bundle:)` / `UIColor(named:in:compatibleWith:)` behind one signature ‚Äî a helper is worth it over 23 `#if os(macOS)` branches because the two APIs diverge in parameter shape (bundle vs in:, optional trait collection on UIKit). Helper traps on missing asset (no graceful fallback) ‚Äî a missing color is a build-authoring error, not a recoverable runtime state. Colorspace `srgb` (closest xcassets match); Mac's pre-refactor `NSColor(red:green:blue:alpha:)` default calibrated-RGB and iOS's `UIColor(red:green:blue:alpha:)` default sRGB are now unified on sRGB ‚Äî visible Mac difference is imperceptible on modern displays. `Theme`'s public API unchanged from the outside: same property names, same types, no consumer code touches needed. SwiftPM's `.process("Resources")` generates `ClearlyCore_ClearlyCore.bundle` automatically; no `project.yml` changes needed.

### 2026-04-21 ‚Äî Collapsed 19 dynamic-color declarations in `Theme`
The original Mac `Theme.swift` had 19 separate `NSColor(name:dynamicProvider:)` declarations. Making each one cross-platform directly in `Theme` would have doubled the rendering file with `#if os(macOS)`/`#else` branches. Instead introduced `PlatformColor.clearlyDynamic(name:light:dark:)` in `Platform.swift` ‚Äî 19 call sites become one-line declarations and `Theme` stays platform-wrapper-only. Same palette, same dynamic behavior.

### 2026-04-21 ‚Äî One `IOSDocumentSession` per iPad tab (Phase 12)
Mac's `WorkspaceManager` shares one editor buffer across tabs via `snapshotActiveDocument` / `restoreActiveDocument`. iOS cannot mirror that ‚Äî `NSFilePresenter` is keyed on URL, and per CLAUDE.md's presenter-lifecycle discipline (add/remove must pair 1:1 with the owning object) every open file needs its own presenter. Giving each `IPadTab` its own `IOSDocumentSession` (and therefore its own presenter) is the only way for remote-change refresh + conflict detection to fire for inactive tabs. Also eliminates the snapshot/restore dance ‚Äî the active tab's session is always already authoritative. Memory cost is small; the ceiling is "how many tabs can the user reasonably open" (roughly 10) √ó (one string + one op queue + one resolver task) per tab.

### 2026-04-21 ‚Äî Tap = open-or-activate, no untitled tabs on iOS (Phase 12)
Rejected the approved plan's "tap replaces active, long-press for new tab" rule in favor of Safari / Bear / Finder semantics: tap always opens-or-activates, long-press context menu is reserved for Rename + Delete. Satisfies the 5-tab success criterion with the simplest interaction model; no "pending-new-tab" flag needed. Also skipped untitled tabs entirely ‚Äî Mac has `createUntitledDocument` but iOS has no scratch folder and we don't want to invent one just for v1 iPad. `‚åòT` opens the quick switcher so "new tab = find or create a file" still works via the existing `VaultSession.openOrCreate` path.

### 2026-04-21 ‚Äî `NavigationSplitView(.balanced)` over manual idiom branching (Phase 12)
iOS 17's auto-collapse handles "3-column on iPad Pro landscape, 2-column on smaller iPads / portrait, collapse to stack on split-screen narrow" without needing `UIDevice.userInterfaceIdiom` checks. Root branch is `@Environment(\.horizontalSizeClass)` at `ContentRoot_iOS`: `.regular` ‚Üí `IPadRootView` (NavigationSplitView), else ‚Üí `SidebarView_iOS` (NavigationStack). Size class reacts to multitasking changes that static idiom would miss; the branch rebuilds the root view but both paths share `FileListView_iOS` + `DocumentDetailBody` so the expensive state (file watcher, open-document sessions, per-tab outlines) lives on `@State` at the app level and survives the rebuild.

### 2026-04-21 ‚Äî WindowGroup over DocumentGroup on iOS (Phase 4)
`IMPLEMENTATION.md` Phase 4 described `DocumentGroup` as the root iOS scene plus a separate welcome `WindowGroup`. Shipped as a single `WindowGroup` instead. `DocumentGroup` on iOS 17 opens the system document browser at launch with one-document-per-scene semantics ‚Äî incompatible with the phase's requirement to show a vault-folder sidebar as the root UI. A single `WindowGroup` hosting welcome + sidebar + detail views matches the Mac app's mental model (Mac uses `Window` + `WorkspaceManager`, not `DocumentGroup`). `MarkdownDocument` was still promoted to `FileDocument` for Phase 5's editor binding. If Files.app "open in Clearly" integration is needed later, `DocumentGroup` can be added as a secondary scene. Phase 12 will revisit scene architecture for iPad multi-tab either way. `IMPLEMENTATION.md` should be updated if this deviation holds through Phase 5.

## Lessons Learned
(What worked, what didn't, what to do differently)
